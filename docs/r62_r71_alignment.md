Each row gives the file, the location (section or line), the current text, and the replacement. Edits are mechanical except where flagged "DECIDE FIRST" (architectural choices captured in /docs/registry/deed.md but not yet reconciled into the older spec text).

| Arweave two-surface consumption model (architectural commitment) | §2.3 Registry, one paragraph after the Master Image / metadata paragraphs | Commit to Arweave's two distinct fetch surfaces: (1) **data body** at `arweave.net/<tx_id>` returns the raw uploaded bytes -- consumed by marketplaces (Phantom / Solana Explorer / Magic Eden) for the Metaplex JSON; (2) **transaction metadata** at `arweave.net/tx/<tx_id>` returns the Arweave protocol wrapper (id, owner, base64-encoded tags, signature) -- consumed by indexers and recovery tooling. This decision rules out (a) "tags-only" metadata schemes, (b) permaweb-SPA translators serving dynamic JSON, (c) `data:` URI hacks pointing at base64 blobs. | Resale architecture: cNFT is transfer-only, SPL Memo carries rotating state (architectural commitment) | §2.3 Registry resale paragraph + §3.5.1 / Card 8 references to `update_metadata_v1`-driven resale re-key | **R62 currently describes resale as: re-upload a new Arweave metadata JSON carrying the rotated `enc_final`, and call `update_metadata_v1` on the cNFT to swap the `uri` to the new tx_id. This is reversed.** The decided architecture, captured in deed.md §2.4: **the cNFT is never updated after mint** -- only Bubblegum `transfer_v2` fires on resale, **composed atomically with an SPL Memo instruction in the same Solana tx**. The Arweave Master file, Arweave metadata JSON, and Arweave legal binder JSON are uploaded ONCE at mint and remain immutable forever. Rotating state (`enc_final` sealed to new owner, `price`, `legal_binder_tx_id`, `monogram`) lives in the Memo payload: `{ "t":"SALE", "p":<price_cents>, "b":<legal_binder_tx_id>, "m":<monogram>, "e":<176-char enc_final> }`. **No custom Solana program. No PDA. No side-car account.** The SPL Memo program is a deployed standard. Discovery recipe for current owner's state: walk Bubblegum transfer log to their tenure-start tx, read the Memo. **Per-resale cost: ~$0.000005 (one Solana base fee for the multi-instruction tx)** -- 200x cheaper than the prior PDA-write architecture (~$0.001). The D-18 seal-break is a **separate** Solana tx fired at first `/download-master` per owner, also using SPL Memo: `{ "t":"UNSEAL", "a":<asset_id>, "u":<enc_final_unwrapped> }`, co-signed by the owner's wallet (INV-02-unforgeable consent to the seal-break). Both Memo patterns work without any custom-program deployment or backfill. INV-02 preserved (Memo bytes are public, but `enc_final` is sealed-box-encrypted to the owner's wallet pubkey; INV-02 says platform never holds buyer privkey, not that ciphertext must be hidden). Keepsake Copy hash (E+N) stays in `Deed.variant_hashes` JSON in the local DB, NOT on-chain, per F-01. The earlier-reserved `DEED_PROGRAM_ID` keypair stays in cold storage as a future option if a richer side-car architecture ever supersedes this; under the current architecture it's unused. |
| Solana-tx ESIGN capture (architectural commitment) | §3.4 Email Delivery + §3.5 Buyer-Retained Evidence Layer (where R62 frames signatures as click events + COA-email-retained evidence) | R62 implicitly treats every ESIGN click (CMA, COA, MJA, SALES_AGREEMENT) as a DB-row + click-metadata capture, with the "buyer-retained evidence layer" of attached PDFs in email serving as the durable trust anchor. **The decided architecture, captured in [cert/esign.md §2.8](cert/esign.md), records each signature as a Solana transaction co-signed by the user's wallet and the platform's `HOT_OPS_KEY`.** The tx carries a Memo with `{t, d, b, p, c}` -- 3-byte document_type code, 5-byte document_id, 43-byte legal_binder_tx_id, per-doc props (450-byte hard cap), and clicked_at. **The user's wallet signature on the tx -- with the wallet privkey held only by them per INV-02 -- is the cryptographically-unforgeable act of consent.** This is strictly stronger than R62's click-wrap framing: click metadata (IP, session token, browser fingerprint) can be fabricated by anyone with platform DB access; a wallet signature on a Solana tx cannot. **Why this matters:** (a) trustless verification -- any future court, auditor, or dispute resolver can independently verify "user X signed binder Y at time T" by querying Solana RPC for ESIGN Memo txs co-signed by X's wallet, without platform DB; (b) post-cessation survival -- the buyer's consent record persists forever on Solana's ledger, independent of our DB, Postmark's audit log, or the buyer's email retention; (c) audit-truth primacy -- per [cert/legal_binder.md §2.4.1](cert/legal_binder.md), the buyer's ESIGN Memo tx is the authoritative record of "which binder did this owner agree to"; the resale-tx Memo is a second independent witness on the same fact. **Cost**: ~$0.000005 per signature (Solana base fee + Memo); platform-paid; ~$0.00002 per deed for the four sig events (CMA, COA, MJA, LICENSE). Negligible at every reasonable scale. **Latency**: ~50-200ms perceived (broadcast acknowledgement); ~1-2 sec confirmation (background sweeper). UX-acceptable for high-stakes one-time signatures (CMA at sign-cma, MJA at first purchase). **Privacy**: click metadata (IP) stays in DB only; the on-chain Memo carries only the 3-byte document_type code, document_id, binder tx_id, props, clicked_at -- no PII on-chain. `session_token_hash` was dropped (provided no value the wallet signature didn't already provide). **MVP scope decision** (per esign.md OI-06): ship Solana-tx capture for **CMA + MJA at MVP** (one-time-per-user lifetime; latency is fine), defer COA + SALES_AGREEMENT (per-image, higher-volume) to post-MVP when latency cost is measured. Click metadata + the binder reference continue to anchor the deferred two at MVP. **Implementation surfaces**: extend `Signature` schema per esign.md §2.10; build the Memo tx in `cert/esign.captureSignature`; have the client (Magic SDK) co-sign via `magic.solana.signTransaction`; broadcast via Solana RPC (Helius in production for sub-slot latency); background `signature_confirmation_sweeper` worker flips status. |
| ISA collapses into COA; COA becomes the signed authenticity document (architectural commitment) | §3.5 R62 legal-artifact paragraphs (where COA is framed as a buyer-retained PDF; ISA terminology doesn't appear in R62 but was introduced in the early MVP spec) | The platform's prior architecture treated ISA (Image Signing Affirmation) and COA (Certificate of Authenticity) as two separate documents: ISA = signed markdown source carrying the creator's authorship attestation; COA = render template producing the buyer-delivered PDF. **The decided architecture collapses these into one COA document** with two presentation forms from the same signed source: (a) the markdown the creator signs at Card 1 (the legal commitment) + (b) the PDF rendered at Card 5 mint by combining the signed markdown with per-deed props (the buyer's delivered certificate). Aligns with traditional art-world COA convention -- COAs are signed by the artist + describe the work + are delivered to the buyer. The "ISA" name was platform-specific jargon for what is essentially a COA. Spec changes: `Signature.document_type` enum drops `IMAGE_SIGNING_AFFIRMATION`, gains `COA` (used as the signed-doc type). `images.signing_event_id_authorship` -> `images.signing_event_id_coa`. Deed metadata JSON: `creator_isa_signing_event_id` -> `creator_coa_signing_event_id`; `creator_isa_signed_at` -> `creator_coa_signed_at`. Binder entries collapse from `{isa, coa}` to a single `coa` markdown entry; the PDF artifact is downstream rendering, not a separate binder slot. Repo file `legal/isa.md` renames + supersedes to `legal/coa.md`. UI fields like `isa_signed_at` on ImageData rename to `coa_signed_at`. |
| SALES_AGREEMENT replaces LICENSE_ACCEPTANCE; signed bilaterally per image (architectural commitment) | §3.5 R62 legal-artifact paragraphs (Title Document / Purchase Receipt / per-image License Acceptance) | R62 treats the per-image use license as a unilateral buyer-side click-wrap ("License Acceptance") with sale terms (price, royalty %, edition) baked into the platform-wide CMA. The decided architecture, captured in [cert/legal_binder.md](cert/legal_binder.md) + [registry/deed.md §1.1](registry/deed.md): **the per-image License Acceptance is replaced by SALES_AGREEMENT, a single per-image document signed by BOTH parties.** Creator signs at Card 1 (bundled with COA under one click; under platform-default terms 10%/1/1 at MVP, per-image variation post-MVP); buyer signs at Card 4 (bundled with MJA on first purchase, alone on subsequent purchases). SALES_AGREEMENT's `content` carries both the commercial terms (price-share, royalty, edition) AND the buyer's use license (display rights, transfer rights, prohibition on commercial reproduction) -- merged because both belong to the same bilateral sale. The COA's `content` is narrowed to authenticity attestation only -- commercial terms move to SALES_AGREEMENT, cleaning the COA's semantic role. Per-image binder props_schema for SALES_AGREEMENT: `{ image_id, royalty_pct, edition, platform_fee_pct }`. Deed metadata JSON gains `creator_sales_agreement_signing_event_id` and `buyer_sales_agreement_signing_event_id` (renamed from `license_signing_event_id`), making the bilateral binding explicit. Schema migration: `purchases.signing_event_id_license` -> `purchases.signing_event_id_sales_agreement_buyer`; new column `images.signing_event_id_sales_agreement_creator`; `Signature.document_type` enum gains `SALES_AGREEMENT`, drops `LICENSE_ACCEPTANCE`. Audit win: "what did the buyer agree to pay royalty on for this deed?" becomes a one-document lookup (the SALES_AGREEMENT both parties signed) instead of CMA + LICENSE_ACCEPTANCE walk. Future-proof: per-image royalty variation, tiered pricing, or negotiated sale terms become trivial -- the SALES_AGREEMENT `p` field carries the values without requiring any schema or binder migration. Repo file `legal/license_acceptance.md` renames + supersedes to `legal/sales_agreement.md` with content merged from the prior license-scope text. |
| Legal binder Arweave permanence (architectural commitment) | §3.4 Email Delivery + §3.5 R62 legal-artifact paragraphs (CMA / MJA / COA / Sales Agreement / Title Document / Purchase Receipt) | R62 frames the COA / Title / Receipt / Sales Agreement PDFs as "buyer-retained evidence layer alongside Postmark's audit log." That's true for the **per-deed evidence** -- but the **canonical contract text** the user agreed to needs its own permanence layer that doesn't depend on the platform's git repo, DB, or email retention. The decided architecture, captured in [cert/legal_binder.md](cert/legal_binder.md): **the platform's signed legal posture (CMA, MJA, COA, SAL) is consolidated into a single legal binder JSON, uploaded to Arweave at MVP launch as one immutable artifact.** A small in-memory cache + local `data/legal_binder.json` file holds the active binder for fast `getActiveBinder()` reads. Every `Signature` row stamps the binder's tx_id on `Signature.legal_binder_tx_id`. Every deed's mint-time Arweave metadata JSON carries the same `legal_binder_tx_id` -- a **single** URI binding all signed contracts for the deed. **Solana-tx ESIGN** (per [esign.md §2.8](cert/esign.md)) records each signature as a Solana transaction co-signed by the user's wallet, with a Memo carrying the `legal_binder_tx_id` -- INV-02-unforgeable consent. **Resale architecture** (post-MVP): each resale's Bubblegum `transfer_v2` Solana tx carries an SPL Memo with the new owner's `legal_binder_tx_id`, price, monogram, and new `enc_final`. Upload cost (one-shot, at MVP launch): ~$0.0005 Arweave (~10-30 KB binder). Recovery story: given any deed's on-chain `uri`, fetch metadata JSON → read `legal_binder_tx_id` → fetch binder JSON → walk `entries[]` to reconstruct exact contract bytes. Zero platform involvement. **MVP scope: one binder ever** -- `npm run release-binder` runs once at launch to create `binder_version: 1`. Binder revisions are post-MVP (see legal_binder.md OI-01). |
